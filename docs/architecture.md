# システム設計書（Architecture）

## 設計意図
このドキュメントは『ビズシスのマーケティングシステムが、どう繋がっているか？』を図と説明で明確にするものです。漢那さんが「ブログを更新したら何が起こる？」「Xに投稿を止めたい場合は？」といった質問に答えられるようになることが目的です。

## 2. データフロー詳細

### 2.1 Astro 実装領域（site/）の位置づけ
- `site/`：Astro プロジェクト全体の実装領域。LP ページ、ブログページ、コンテンツ、ビルド設定を収める。
- `site/src/pages/`：LP用ページとブログ用ページを配置するルート。
  - `site/src/pages/index.astro`：ホーム
  - `site/src/pages/about.astro`：会社紹介
  - `site/src/pages/services.astro`：サービス紹介
  - `site/src/pages/cases.astro`：導入事例
  - `site/src/pages/tools.astro`：利用ツール・強み
  - `site/src/pages/faq.astro`：よくある質問
  - `site/src/pages/contact.astro`：問い合わせフォーム
  - `site/src/pages/blog/index.astro`：ブログ一覧
  - `site/src/pages/blog/[slug].astro`：ブログ個別記事
- `site/src/content/blog/`：Markdown 記事の保存先。URL は `/blog/[slug]` 形式で公開される。
- `site/src/components/`（必要に応じて）：共通ヘッダー・フッター・CTA など再利用コンポーネントを配置する。
- 現時点では `site/src/pages/index.astro` が Astro 初期化のプレースホルダになっている。実装担当はこのファイルを本構成に合わせて置き換える。
- GitHub Pages の公開ブランチは `main`（または運用で指定したブランチ）を想定し、Astro は `dist/` へのビルド出力を公開用ファイルとして扱う。
- `CNAME` は `site/public/CNAME` に `biz-sys.jp` を記載するか、GitHub Pages のカスタムドメイン設定で管理する。

#### 各ページの役割と主要コンテンツ
| ページ | 役割 | 主要コンテンツ |
|--------|------|---------------|
| `/` | LPトップ | 事業概要、USP、CTA、主要サービスへのリンク |
| `/about` | 会社紹介 | ビズシスのミッション・強み・実績 |
| `/services` | サービス紹介 | 提供サービス一覧、特徴、導入効果 |
| `/cases` | 導入事例 | 事例紹介、成果、顧客の声 |
| `/tools` | 利用ツール | 使用するツール・自動化の仕組み説明 |
| `/faq` | よくある質問 | よくある疑問と回答、安心材料 |
| `/contact` | 問い合わせ | Formspree フォーム、連絡先、相談の流れ |
| `/blog/` | ブログ一覧 | 最新記事リスト、人気記事、カテゴリ紹介 |
| `/blog/[slug]` | 個別ブログ | 記事本文、CTA、関連記事へのリンク |

### 2.2 output/ と site/ の役割分離
- `site/`：実装担当が編集・開発する開発領域。Astro のソースコード・テンプレート・コンテンツを含む。
- `output/`：ビルド済みの成果物や公開用ファイルを格納する領域。手動提出や成果確認のために保持する。
- 実装担当は `site/` の構成を確定した上でページ追加を進め、完成後に必要に応じて `output/` に生成物を格納する。

### 2.3 Formspree 問い合わせフォームの想定配置
- 掲載ページ: `/contact`
- 想定項目:
  - お名前
  - 会社名 / 団体名
  - メールアドレス
  - お問い合わせ内容
  - 相談希望日 / 備考
- 送信先: 漢那さんのメールアドレス
- 運用確認: テスト送信で受信できることを確認し、スパムフォルダもチェックする。

## 3. 主要コンポーネント＆役割

### 3.1 GitHub Repository（ソース管理）
| 項目 | 説明 | 漢那さんの操作 |
|------|------|---------|
| **役割** | HP・ブログ・自動化スクリプトの一括管理 | docs/blog/ に記事を追加・更新 |
| **所在** | bizsys-pj（非公開リポジトリ） | GitHub Desktopで Commit & Push |
| **重要ファイル** | docs/blog/、automation/、_config.yml | 記事ファイルのみ編集 |
| **バックアップ** | GitHub が自動で保持（コード履歴まで） | 削除で困ることはほぼなし |

### 3.2 GitHub Pages（HP・ブログ公開）
| 項目 | 説明 | 漢那さんの操作 |
|------|------|---------|
| **役割** | ブラウザから見える『公開サイト』 | 閲覧のみ（編集は不可） |
| **所在** | biz-sys.jp (GitHub Pages経由) | サイトが正しく表示されるか確認 |
| **更新方法** | GitHub Repository の push 時に自動更新 | 自動なので操作不要 |
| **障害時** | GitHub Pages が一時的に落ちることはほぼなし | 1時間以上つながらないときは報告 |

### 3.3 GitHub Actions（自動デプロイ・投稿）
| 項目 | 説明 | 漢那さんの操作 |
|------|------|---------|
| **役割** | push → 自動ビルド → 自動投稿 | 何も操作不要（自動実行） |
| **トリガー** | docs/blog/ に新しい記事 push 時 | - |
| **実行時間** | 2～3分 | push 後、待つだけ |
| **ログ確認** | GitHub リポジトリの`Actions`タブで確認 | エラー時は確認推奨 |

### 3.4 X（Twitter）API連携
| 項目 | 説明 | 漢那さんの操作 |
|------|------|---------|
| **役割** | GitHub Actions → @bizsys_jp に自動投稿 | 何も操作不要（自動実行） |
| **認証方式** | OAuth認証（安全な方式）| アカウント名・パスはGitHub Secretsに暗号化保存 |
| **失敗時** | X API障害 or 認証期限切れ | operation.md 参照 |
| **手動投稿** | 必要時は手動で @bizsys_jp へ投稿可 | Twitter公式アプリから通常通り投稿 |

### 3.5 Google Analytics 4（計測）
| 項目 | 説明 | 漢那さんの操作 |
|------|------|---------|
| **役割** | HP訪問数・ブログPV・訪問者行動の記録 | GA4ダッシュボードで月1回確認 |
| **データ保持** | 無制限（Google が管理） | 削除不可（意図的に消さない限り） |
| **計測対象** | biz-sys.jp のすべてのページ | 自動計測（操作不要） |
| **プライバシー** | 個人情報は保存しない（アクセス統計のみ） | IP匿名化設定済み |

### 3.6 Formspree（問い合わせフォーム）
| 項目 | 説明 | 漢那さんの操作 |
|------|------|---------|
| **役割** | HP の問い合わせフォーム受信 | 漢那さんのメール自動受信 |
| **送信先** | 漢那さんのメールアドレス | 問い合わせメールに返信 |
| **費用** | 無料（月50件まで） | 超過時は有料プランへ切り替え |
| **スパム対策** | 自動フィルタ有り | spam フォルダを月1回確認 |

## 4. 依存関係＆リスク

### 4.1 外部サービス依存関係

```
┌─────────────────────────────────────────────────┐
│  ビズシスシステムが依存している外部サービス      │
├─────────────────────────────────────────────────┤
│ ◆ 【必須】GitHub                              │
│   → gitlab/Gitea等では代替可能（設定変更要） │
│                                                │
│ ◆ 【必須】GitHub Pages                        │
│   → Netlify/Vercel等では代替可能              │
│                                                │
│ ◆ 【必須】Astro（静的サイト生成）            │
│   → Hugo/Jekyll等では代替可能（設計変更要） │
│                                                │
│ ◆ 【重要】X API                               │
│   → 止まった場合は手動投稿に切り替え         │
│                                                │
│ ◆ 【重要】Google Analytics 4                 │
│   → Matomo等で代替可能（移行時間 1日程度）  │
│                                                │
│ ◆ 【便利】Formspree                          │
│   → メール受信のみなので代替が簡単           │
└─────────────────────────────────────────────────┘
```

### 4.2 障害時の対応（簡易版）

| 障害シナリオ | 対応方法 | 復旧時間 |
|-----------|--------|--------|
| **GitHub Pages がダウン** | 通常 1時間で復旧 / それ以上の場合は Netlify へ緊急移行 | 1～2h |
| **X API が通信不可** | GitHub Actions がエラー / 手動で @bizsys_jp へ投稿 | 2h～ |
| **GA4 が計測できない** | 設定確認（operation.md 参照） | 0.5h |
| **GitHub リポジトリ削除** | バックアップリポジトリから復旧 | 1h |

## 6. スケーラビリティ＆拡張性

### このシステムで増やせること

```
✅ 記事数を増やす
   → 毎週5本まで余裕（GitHub Pages 無制限）

✅ SNS投稿を増やす
   → X以外を追加する場合は automation/ に新しいワークフロー

✅ アクセス者を増やす
   → GA4 は無制限、GitHub Pages も無制限

✅ トラフィックを増やす
   → GitHub Pages は月間5GB まで無料

✅ 自動化を強化する
   → Claude Code で記事生成スクリプトを追加可能
```

### このシステムで難しいこと

```
✗ Astro 以外のフレームワークに乗り換える
   → 設計全体の見直しが必要 ⇒ 簡単ではない

✗ 複雑な会員機能・ショッピング機能を追加
   → Static Site Generator（GitHub Pages）では対応困難
   → 別の仕組みが必要（コスト増）

✗ リアルタイム更新（チャット・ライブ通知）
   → GitHub Pages + Static では不可
   → 別サーバーが必要
```

## 7. 非エンジニア向けQ&A

**Q. Markdown って何？**  
A. 『プレーンテキスト（メモ帳）で書く簡単なフォーマット』。HTML を直接書く必要はありません。

**Q. GitHub って何？**  
A. 『プログラムのバージョン管理＆共有ツール』。Dropbox の『編集履歴版』みたいなものです。

**Q. GitHub Actions って何？**  
A. 『push したら自動で実行されるロボット』。記事を追加したら自動でサイトを作り変えて、X にも投稿します。

**Q. GitHub Secrets って何？**  
A. 『GitHub 上の『金庫』』。X API キーなど大事な情報を暗号化して保管します。見られることはありません。

**Q. もし何か壊れたら？**  
A. GitHub の履歴から『昨日の状態に戻す』が簡単にできます。ご安心を。
