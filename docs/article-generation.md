# ブログ記事生成ガイド — Claude Code を使った下書き作成

> 対象：漢那さん  
> 更新日：2026-04-10

---

## 1. 概要

このドキュメントは、Claude Code を使ってビズシスのブログ記事の下書きを作成するための手順書です。

プロンプトテンプレートをコピーして Claude Code のチャットに貼り付けるだけで、`site/src/content/blog/` にそのまま置ける形式の記事下書きが生成されます。

---

## 2. ビズシスブログの記事要件

### ターゲット読者

- 中小企業の経営者・管理職
- 非エンジニア。IT リテラシーは低め
- 「Excel での管理に限界を感じているが、何から始めれば良いかわからない」層

### トーン

- 親しみやすく、専門的すぎない
- 「難しそう」という印象を取り除く
- 結論→理由→具体例の順で書く（忙しい経営者が流し読みできる）

### 記事構成（2026-06-19更新）

```
## はじめに
（読者の課題を提示 + 読むメリット箇条書き）

## 課題の具体例
（「あるある」な状況の描写）

## 解決策の提示
（ビズシスのアプローチ・具体的なステップ）

## システム化でできること（主な機能）
（具体的な機能名を箇条書き。機能名がイメージできる粒度で）

## 小さく始めるなら、どの機能から
（一番手間がかかる業務ひとつを名指し。FAQと重複させない）

## 相談前に準備しておくとスムーズなもの
（Excel/帳票サンプル・困っている業務・規模感・予算希望。ハードルを上げない）

## よくある質問
（FAQ）

## まとめ・次のアクション
（要約 + 問い合わせへの誘導 + 関連ページ導線）
```

- **本文 3,500文字程度**（2026-06-19に3,000→3,500へ引き上げ）
- **まとめ末尾の関連ページ導線は内部リンクを相対パスで書く**（`/services` `/cases` `/demo` `/pricing`）。事例リンクは記事テーマに近い事例がある時だけ。
- 詳細は `docs/templates/blog_template.md` と memory `project-blog-spec` を正とする。

### frontmatter 仕様

```markdown
---
title: "記事タイトル（SEOキーワードを含める）"
date: YYYY-MM-DD
tags: [タグ1, タグ2]
icon: "icon-gears.png"
author: "ビズシス"
description: "記事の説明（70〜90文字程度。X自動投稿にも使用されるため、X投稿として自然な文章にする）"
---
```

**`icon:` に指定できる画像ファイル一覧**（`/public/images/blog/` に格納）

| ファイル名 | 用途イメージ |
|---|---|
| `blog-thumb-default.png` | **汎用サムネ（初回はこれを設定する）** |
| `thumb-excel-exit.png` | Excel脱却・注目記事系 |
| `icon-gears.png` | 自動化・DX |
| `icon-people-network.png` | 属人化・チーム |
| `icon-clipboard.png` | チェックリスト・発注 |
| `icon-chart-up.png` | 効率化・メリット |
| `icon-handshake.png` | 会社選び・パートナー |
| `icon-cycle.png` | 保守・運用・継続 |
| `icon-boxes.png` | 在庫管理・Excel管理 |
| `icon-shield-check.png` | セキュリティ・DX入門 |

> **ルール：** 新しいブログ記事を作成した初回は必ず `icon: "blog-thumb-default.png"` を設定する。
> 記事に合うアイコンが決まったタイミングで上記一覧の中から適切なものに差し替える。
> `icon:` を省略した場合は `icon-gears.png` がフォールバックで表示される。

### ファイル命名規則

- 形式：`YYYY-MM-DD-記事スラッグ.md`
- スラッグはローマ字・英数字・ハイフンのみ
- 例：`2026-04-15-excel-kanri-kaiketsu.md`
- 保存先：`site/src/content/blog/`

## 4. 使い方手順（ステップ・バイ・ステップ）

### Step 1：Claude Code を開く

VS Code のターミナルで以下を実行するか、Claude Code のアイコンをクリックして起動します。

```bash
claude
```

### Step 2：プロンプトを貼り付けて記事を生成する

1. 上記「3. プロンプトテンプレート」のコードブロック内の文章を**全文コピー**する
2. Claude Code のチャット入力欄に貼り付ける
3. `[　]` の項目を書き換える（キーワード、タイトル案、日付など）
4. Enter で送信する

### Step 3：生成内容を確認する

Claude Code が生成した Markdown（` ```markdown ` ブロック内）を確認します。
問題があれば「〇〇を修正してください」と追加で指示を送ります。

### Step 4：ファイルに保存する

1. `site/src/content/blog/` フォルダを開く
2. 推奨ファイル名（例：`2026-04-15-excel-zaiko-kanri-kaiketsu.md`）で新規ファイルを作成する
3. 生成された ` ```markdown ` ブロックの**中身だけ**（` ```markdown ` と ` ``` ` の行を除いた部分）を貼り付けて保存する

### Step 5：push して公開する

ターミナルで以下を実行します：

```bash
cd c:\Dev\bizsys-pj
git add site/src/content/blog/（作成したファイル名）
git commit -m "feat: ブログ記事を追加 — （記事タイトル）"
git push origin main
```

push 後、GitHub Actions が自動でサイトをデプロイし、X にも自動投稿されます。

## 5. 生成品質のチェックポイント

push 前に以下を確認してください。

| チェック項目 | 確認方法 |
|------------|---------|
| frontmatter が正しく記述されているか | `---` で囲まれた部分がファイルの先頭にあるか確認 |
| `title` に SEO キーワードが入っているか | 読者が検索しそうな言葉が含まれているか |
| `description` が 70〜90 文字か | 文字数をコピーして確認（短すぎ・長すぎに注意） |
| `description` が X 投稿として自然か | 読んで違和感がないか |
| `date` が正しい日付形式か | `YYYY-MM-DD`（例：`2026-04-15`） |
| `icon` が指定されているか | 上記アイコン一覧から内容に合うファイル名を選んで記述 |
| `author` が `"ビズシス"` になっているか | そのまま確認 |
| ファイル名がルール通りか | `YYYY-MM-DD-スラッグ.md`、スラッグはローマ字・英数字・ハイフンのみ |
| 記事が規定の構成になっているか | はじめに／課題／解決策／**機能一覧／小さく始める／相談前準備**／FAQ／まとめ |
| 本文が3,500文字程度か | 空白・関連記事を除いてカウント |
| まとめに問い合わせへの誘導があるか | `/contact` へのリンクまたは言及 |
| まとめ末尾に関連ページ導線があるか | サービス/事例/デモ/料金（**相対パス**。事例は近い事例がある時だけ） |

## よくあるトラブル

**Q. 生成内容が長すぎる・短すぎる**  
A. 「はじめに を 100〜150 文字に収めてください」のように、セクション単位で修正を依頼してください。

**Q. description が 90 文字を超えてしまう**  
A. 「description を 80 文字以内に短くしてください」と追加指示を出してください。

**Q. X 投稿の文章として不自然**  
A. 「description を X（Twitter）の投稿として自然な文体に書き直してください」と依頼してください。
