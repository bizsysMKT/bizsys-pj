## 1. 記事追加の流れ（必ずこの順序で行う）

> いきなり記事作成に入らない。テーマは必ずデータで決める。

1. **GA4 + Search Console 分析**：現状の表示・クリック・順位を確認する（月次分析の手順は `docs/monthly-analysis.md`）
2. **（必要なら）キーワード調査**：`python -X utf8 automation/keyword-research.py` でGoogleサジェスト＋SCの実需を確認する（手順は `docs/keyword-research.md`）
3. **テーマ決定**：後述「テーマ選定の優先順位」に従い、データの裏付けがあるテーマだけを選ぶ
4. **記事作成**：本ガイドの要件・ルール・テンプレート（後述）に従って `site/src/content/blog/スラッグ.md` を作成
5. **公開前チェック**：後述「公開前チェック」の2つの機械チェック＋チェックリストを実施
6. **コミット・push**：コミットメッセージは `【追加】〜` 形式。push すると GitHub Actions が自動デプロイする（未来日付の記事は予約投稿となり、日付到来日の深夜0時の定期ビルドで自動公開される）

## 2. 記事の基本要件

### ターゲット読者

- 中小企業の経営者・管理職
- 非エンジニア。IT リテラシーは低め
- 「Excel での管理に限界を感じているが、何から始めれば良いかわからない」層

### トーン

- 親しみやすく、専門的すぎない
- 「難しそう」という印象を取り除く
- 結論→理由→具体例の順で書く（忙しい経営者が流し読みできる）

### 文字数

- **本文3,500文字程度**（空白・関連記事を除く。2026-06-19に3,000→3,500へ引き上げ）

### 記事構成（この順序で書く・2026-06-19更新）

```
## はじめに
（読者の課題を提示 + 「この記事を読むとわかること」箇条書き）

## 課題の具体例
（「あるある」な状況の描写）

## 解決策の提示
（システム化で変わること + 「### 具体例：〜のイメージ」）

## システム化でできること（主な機能）
（具体的な機能名を箇条書き。機能名がイメージできる粒度で）
※監査ログ的機能は書かない（後述の注記参照）

## 小さく始めるなら、どの機能から
（一番手間がかかる業務ひとつを名指し。FAQと重複させない）

## 相談前に準備しておくとスムーズなもの
（Excel/帳票サンプル・困っている業務・規模感・予算希望。ハードルを上げない）

## よくある質問
（FAQ。費用の質問には料金ページの数値で回答）

## まとめ・次のアクション
（要約 + 問い合わせへの誘導 + 関連ページ導線カード）
```

- 本文中盤に「**○○しやすい業務例**」の `.biz-links` カード（後述）を入れる
- **Before/After比較表を1つ**入れる（推奨）
- 関連記事を**3本以上**、記事末尾にリストアップする

### まとめ末尾の関連ページ導線（`.page-links` カード）

markdownの箇条書きではなく、次のHTMLで書く（相対パス・別タブ）。**事例（/cases）リンクは記事テーマに近い事例がある時だけ**置く。サービス/デモ/料金は常時OK：

```html
<p class="page-links-lead">あわせて、こちらもご覧ください。</p>

<div class="page-links">
<a target="_blank" rel="noopener" href="/services/"><b>サービス内容</b>システム開発の概要・流れ</a>
<a target="_blank" rel="noopener" href="/cases/"><b>導入事例</b>他社の改善イメージ</a>
<a target="_blank" rel="noopener" href="/demo/"><b>操作デモ</b>実際の画面の動き</a>
<a target="_blank" rel="noopener" href="/pricing/"><b>料金</b>費用の目安</a>
</div>
```

> リンク先URLは必ず末尾スラッシュ付き（後述「4. 内部リンクは必ず末尾スラッシュ付きで書く」参照）。

## 3. 集客記事として強くする方針（2026-06-30追加）

ChatGPTレビューを受けて合意した、SEO記事を「商談につながる記事」に近づける方針。汎用的に効くものだけルール化。

- **一般論を減らし、具体例・失敗例・改善策で書く（この方針の土台）**
  - 「〜が大切です」「〜という課題があります」で終わらせない。**具体的な業務シーン → あるある失敗例 → それをどう改善するか**をセットで示す。読者が自分ごと化でき、商談につながりやすい。各記事の「### 具体例：〜のイメージ」もこの一環。
  - **既存ルールと両立させる（ここがズレやすいので注意）**：
    - ①トーンは崩さない＝「親しみやすく・専門的すぎない・非エンジニア向け」のまま。**具体≠専門用語**（DB設計やコード詳細に踏み込まない。あくまで業務の具体）。
    - ②改善策は**「何が変わるか」の粒度**に留める。具体的にしようとして重い機能（変更履歴＝監査ログ等）を約束しない（後述の監査ログ禁止ルール参照）。
    - ③本文3,500字を膨らませない＝一般論を**"足す"のではなく具体例に"置き換える"**。

- **「○○しやすい業務例」セクションを本文中盤に入れ、既存記事へ内部リンクする（最重要）**
  - そのテーマで属人化・非効率になりやすい**具体業務（受注管理・在庫管理・案件管理・配車・生産管理・見積・営業日報・顧客管理 等）**を列挙し、各業務を対応する既存ブログ記事へ相対リンクする。
  - **出力は次のHTMLで書く**（カード表示はCSSの `.biz-links` が担当。markdownの箇条書きではなくこの形にする）：
    ```html
    <div class="biz-links">
    <a target="_blank" rel="noopener" href="/blog/対象スラッグ/"><b>業務名</b>対象記事のタイトル</a>
    <a target="_blank" rel="noopener" href="/blog/対象スラッグ2/"><b>業務名2</b>対象記事のタイトル2</a>
    </div>
    ```
    `<b>` が業務名ラベル（1段目）、続くテキストが記事タイトル（2段目）。タイトルはリンク先記事の実タイトルにする。**`target="_blank" rel="noopener"` を付け別タブで開く**（読者が記事を読む途中で開いても読み位置を失わないため）。
  - 狙い：①読者の自分ごと化（「あ、うちのことだ」）②記事クラスタを内部リンクで"面"にしてSEOを底上げ③一般論から「業務システム化」文脈への橋渡し。
  - 抽象テーマ（属人化・DX・業務改善など）の記事ほど効果が大きい。テーマに合う業務だけ挙げる（無関係な業務を並べない）。
- **Before/After比較表を1つ入れる（推奨）**
  - 「現状（手作業・属人化）」vs「システム化後」を2〜4行の表で示し、文章だけの単調さを減らす。抽象テーマで特に有効。
  - 比較表には**明確な差だけ**を載せる。ニュアンス込みの論点（保守は開発会社が対応する等）は表に入れず本文で扱う。
- **タイトルは検索の軸キーワードを据え置く（注意）**
  - Search Consoleで表示が出ている軸キーワード（例：`中小企業 属人化`）を、装飾語で薄めたり言い換えたりしない。Excel・業務システム文脈は**タイトルではなく本文（業務例セクション等）で自然に**足す。本命キーワードのマッチを崩すと順位を落とすリスクがある。
- **「テーブルの変更前後を保持する監査ログ系機能」だけを想起させない（2026-06-30追加）**
  - **避けるべきはこれ1点だけ**：読者に「**テーブルの変更前後（紐づく関連テーブルの変更も含めて）を保持し、いつ・誰が・どう変えたかの差分がまるごと残る／たどれる**」という監査ログ系の機能を想起させること。これは全テーブルの変更前後＝差分を保持する仕組みが必要で実装コストが高く、安易に約束すると見積・実装で大きなギャップになる。
  - 機能一覧（## システム化でできること）に書かないのはもちろん、**課題セクションの本文でも「Excelは変更の経緯が残らない＝システムなら残せる」と匂わせない**（暗にこの監査ログを約束することになる）。
  - **逆に、これさえ想起させなければOK。** 「対応履歴・商談履歴・日報の記録」「見積の改訂版管理」「いつ・誰が・何を入力／作業したかの記録」のように**ユーザーが入力したデータを時系列で残すだけのもの**は実装が軽いので問題なく書いてよい。判断軸は「データの変更前後（差分）を自動で追跡するか」——する＝NG、入力を残すだけ＝OK。迷ったら書かない。
- **（将来課題・未ルール化）記事別CTA**：CTA文言を記事テーマに寄せると問い合わせ率は上がるが、現状CTAは共有コンポーネント（`blog/[slug].astro`・Layout）に直書き。frontmatterにCTA文言フィールドを足すテンプレ改修が必要で、1人運用方針とのトレードオフがあるため、引き継ぎ方針と相談してから判断する。

## 4. 禁止・必須ルール集

### 内部リンクは必ず「末尾スラッシュ付き」で書く（2026-07-08追加・最重要）

記事本文からサイト内へ張るリンク（`.biz-links`・`.page-links`・「関連記事」リスト・本文中のリンクすべて）は、**必ずURLの末尾にスラッシュ `/` を付ける**。

- ✅ 正しい：`href="/blog/zaiko-kanri-ai-jisaku/"`、`[記事](/blog/zaiko-kanri-ai-jisaku/)`、`href="/pricing/"`
- ❌ 誤り：`href="/blog/zaiko-kanri-ai-jisaku"`、`[記事](/blog/zaiko-kanri-ai-jisaku)`、`href="/pricing"`

**なぜ必須か：** サイトは `trailingSlash: 'always'`（スラッシュ付きが正規URL）で運用している。スラッシュ無しで書くと、同じ記事が `/blog/foo` と `/blog/foo/` の**2つのURLとしてGoogleに二重インデックスされ、SEO評価（表示回数・被リンク）が分散して順位が上がらなくなる**。実際2026-07-08に、記事本文222本の内部リンクがスラッシュ無しで混在していたのを全面修正した（`barcode-zaiko-kanri` が2URLに割れて表示が66/37に分裂していた）。

- 対象は `/blog/スラッグ/` だけでなく `/services/` `/cases/` `/demo/` `/pricing/` `/contact/` などLPページへのリンクも同じ（すべてスラッシュ付き）。
- `/blog/tag/タグ名/` も同様にスラッシュ付き。
- 例外：ページ内アンカー（`#見出し`）と外部URL（`https://〜`）はそのまま。
- 公開前チェックの機械チェックで残存を検出できる（後述「7. 公開前チェック」参照）。

### 商売上マイナスになる記述の禁止（2026-07-04追加）

費用・保守系記事のレビューで以下が繰り返し削除された。今後は最初から書かない：

- **既製品・クラウドツールへの誘導**（「既製品で足りるか先に確認」「月額ツールで済む場合も」等）→ 他社サービスに流す記述はNG
- **解約・契約終了の話題**（「保守契約は途中でやめられるか」等）→ 解約をイメージさせるFAQ・記述はNG
- **読者に交渉材料を与える記述**（「実費と作業費の内訳で妥当性を判断」「線引きを契約前に確認せよ」等）→ 内訳開示・確認行動の推奨はNG
- **約束しすぎる保守範囲**（OS/ミドルウェアのアップデート対応等）→ 保守内容はサーバー維持・バックアップ・不具合対応・軽微な調整・SSL更新までに留める
- ヒアリング項目（相談前に準備しておくとスムーズなもの）に「業務停止時の影響度」のような重い質問を入れない

**趣旨：** 記事は集客資産であり、読者への中立的アドバイスより「ビズシスへの相談につながるか」を優先する。事実を偽るのではなく、**読者を他社・自前解決・値下げ交渉へ誘導する記述を書かない**ということ。

### 技術前提：バーコード読み取りの書き方（2026-07-06追加）

在庫・現場系の記事でスキャン運用に触れるときは、次の技術前提で書く（実運用の実感にもとづく）：

- **QRコード＝スマホカメラで安定して読める**と書いてよい（JSライブラリでも実用レベル）
- **1次元バーコード（JANコード等）＝ブラウザ/スマホカメラでは精度が実用レベルに届きにくい**。「スマホのカメラでバーコードが読める」とは書かない
- 1次元バーコードの現実解は**外付けバーコードリーダー**：Bluetooth/USB接続・**1台数千円〜2万円程度**・読んだ値がそのままキーボード入力として入る仕組みで、Webシステムとの相性が良い。精度・速度は専用ハンディと遜色ない、と書く
- 自社発行ラベルの現場には「**QRコードで発行すればカメラ運用できる**」を提案の幅として添える

### 表現ルール：強い断定表現をしない（2026-07-06追加）

- 「**必ず**〜になる」「**絶対に**〜する」「〜は**100%**崩れる」のような強い断定は書かない（レビューで「表現が強い」と修正指示が入った実績あり）
- 言い換えの例：「必ず崩れます」→「崩れていき**がちです**」／「必ず起きます」→「**多くの場合**起きます」「起きる**ことがよくあります**」
- 理由：①事実として言い切れないことを断定すると信頼を落とす ②読者の状況は様々で、当てはまらない読者には押し付けに見える
- 効果・改善の約束も同様（「半減します」→「半減した事例もあります」「〜が目安です」等、実態の範囲で書く）

### 表現ルール：AI自作系記事で「引き継ぐ」と書かない（2026-07-02追加）

- AIや自作で作りかけたシステムについて、「プロが**引き継ぐ**」という表現は使わない。**「作り直す」のみ**を謳う（他人・AIが書いたコードの引き継ぎ・保守は引き受けない方針のため）
- FAQの質問文も「引き継いでもらえますか？」ではなく「作りかけたものは無駄になりますか？→無駄になりません。要件整理の材料として活かし、作り直せます」の形にする
- ※「既存データの引き継ぎ」（データ移行の意味）はOK。禁止はコード・システムの引き継ぎ表現のみ

### 料金ページとの整合（費用に触れる記事の必須ルール）

費用・料金に触れる記事は、必ず `/pricing` の数値と一致させる：

| 区分 | 金額 |
|---|---|
| Excel脱却・一元管理 | 40万円〜 |
| 業務効率化（集計・帳票） | 80万円〜 |
| オーダーメイド開発 | 100万円〜 |
| 保守 | 月1万円〜 |

FAQの費用回答もこの数値で統一し、[料金ページ](/pricing/)への相対リンクを添える（末尾スラッシュ付き）。

### テーマ選定の優先順位（2026-07-04追加）

記事テーマは「受注までの距離」が近い順に選ぶ：

1. **業界×デモ系**（「○○業 業務システム販売管理 デモ」等）＝導入を具体検討中の層
2. **エクセル限界・費用系**（「○○ エクセル 限界」等）＝課題が顕在化した層
3. **導入手順系**（「バーコード 在庫管理 導入」等）＝改善手段を物色中の層

**「○○ 自作」「AI自作」系は増産しない**（検索者は「これから自作したい人」で発注意欲が低い。既存のAI自作記事は"自作に挫折したときの受け皿"として維持するのみ）。費用系も量産しない（キーワード調査で需要薄と確認済み。2026-07-04）。テーマは必ずGA4+Search Console分析→（必要なら）キーワード調査を経てデータで決める。

## 5. frontmatter 仕様・ファイル命名規則

### frontmatter 仕様

```markdown
---
title: "記事タイトル（SEOキーワードを含める・40字程度・できるだけ具体的な数値を入れる）"
date: YYYY-MM-DD
tags: [タグ1, タグ2, タグ3]
icon: "blog-thumb-default.png"
author: "ビズシス"
description: "記事の説明（70〜90文字。具体的な数値を入れる）"
---
```

> **タイトル・descriptionの方針：できるだけ具体的な数値を入れる（2026-06-30追加）**
>
> 「具体ステップ」より「**具体的な4ステップ**」、「複数の機能」より「**6つの機能**」のように、
> ステップ数・機能数・件数・年数・割合などの**具体的な数値**を入れると、記事の中身が一目で伝わり検索結果でのクリック率（CTR）が上がりやすい。
> - **タイトルに優先して入れる**（検索結果で最初に目に入るため効果が大きい）。続けてdescriptionにも入れる。
> - 記事本文の実態と一致する数値にする（4ステップと書くなら本文も4ステップ）。
> - **タイトルとdescriptionは同じ数値で揃える**（例：タイトル「具体的な4ステップ」→descriptionにも「4ステップ」、本文H2見出しも「4ステップ」）。
> - 数値は半角（`4` `6` `80万円`）で書く。
> - 数値が自然に入れられないテーマで無理に作らない。入れられる場合は優先して入れる。

**`icon:` に指定できる画像ファイル一覧**（`/public/images/blog/` に格納）

| ファイル名 | 用途イメージ |
|---|---|
| `blog-thumb-default.png` | **汎用サムネ（初回はこれを設定する）** |
| `thumb-excel-exit.png` | Excel脱却・注目記事系 |
| `icon-gears.png` | 自動化・DX |
| `icon-people-network.png` | 属人化・チーム |
| `icon-clipboard.png` | チェックリスト・発注 |
| `icon-chart-up.png` | 効率化・メリット |
| `icon-handshake.png` | 会社選び・パートナー |
| `icon-cycle.png` | 保守・運用・継続 |
| `icon-boxes.png` | 在庫管理・Excel管理 |
| `icon-shield-check.png` | セキュリティ・DX入門 |

> **ルール：** 新しいブログ記事を作成した初回は必ず `icon: "blog-thumb-default.png"` を設定する。
> 記事に合うアイコンが決まったタイミングで上記一覧の中から適切なものに差し替える。
> `icon:` を省略した場合は `icon-gears.png` がフォールバックで表示される。

### ファイル命名規則（2026-06-30更新：日付なしスラッグへ移行）

- **形式：`記事スラッグ.md`（ファイル名に日付を入れない）**
- スラッグはローマ字・英数字・ハイフンのみ（スペース・日本語はNG）
- 例：`excel-zaiko-kanri-kaiketsu.md` → URLは `/blog/excel-zaiko-kanri-kaiketsu`
- 保存先：`site/src/content/blog/`

> **なぜ日付をファイル名から外したか（2026-06-30）**
>
> 公開URLは「ファイル名から `.md` を除いたもの」になる（`blog/[slug].astro` の `getStaticPaths`）。
> 旧形式 `YYYY-MM-DD-スラッグ.md` だと **URLに日付が露出し、数年後に古く見える／evergreen記事をリフレッシュしてもURLが当時の日付のまま** という弱点があった。
> 一方、**予約投稿フィルタ・並び順・関連記事はすべて frontmatter の `date:` を見ており、ファイル名の日付は一切参照していない**（コード確認済み）。
>
> - **新規記事は必ず日付なしスラッグで作る**。`redirects` への追記は不要
> - 公開日の制御（ドリップ公開含む）は必ず frontmatter の `date:` で行う
> - 2026-06-30に旧33本も日付なしへ統一済み。旧URLは `site/astro.config.mjs` の `redirects` で新URLへ転送している（GitHub Pages配信のため `meta refresh`＋`canonical`＋旧ページ`noindex` の静的リダイレクト）
> - 旧記事のスラッグを今後さらに変える場合は、同様に `redirects` に1行追加して旧URL→新URLを維持する

### 予約投稿の仕組み（重要）

本番サイトでは「記事の日付（`date`）が**今日以前**のものだけ」が公開される。**未来の日付にした記事は、pushしてもすぐには公開されず、その日付になると深夜0時の定期ビルドで自動公開される（予約投稿）。**

- **すぐ公開したい** → `date` を今日以前にする
- **順次公開したい** → 複数記事を未来の日付で1日ずつずらす（1日1本のドリップ公開）
- devサーバー（`npm run dev`）では未来日付の記事も表示される。除外されるのは本番ビルド時のみ
- **記事を追加したのにサイトに出てこないときは、まず `date` が未来になっていないか確認**（不具合ではなく仕様）
- 実装箇所：`site/src/pages/blog/index.astro` の日付フィルタ（`blog/[slug].astro`・`blog/tag/[tag].astro` にも同じフィルタあり）

## 7. 公開前チェック（セルフレビュー必須）

### 機械チェック2点（2026-07-02追加）

1. **太字のレンダリング確認**：日本語では `文字**「〜」**文字` のように**太字の境界にカギ括弧（「」）や〜があり、外側が通常文字**だと、CommonMarkの仕様で太字にならず `**` がそのまま画面に表示される。安全な書き方は「括弧を太字の外に出す（`「**〜**」`）」「`**`の直前に読点を置く」「太字の端を通常文字にする」のいずれか。確認コマンド（devサーバー起動中に実行。何も出なければOK）：
   ```bash
   curl -s http://localhost:4321/blog/スラッグ | grep -o '.\{25\}\*\*.\{25\}'
   ```
2. **内部リンクの公開順チェック**：予約投稿の仕組み上、リンク先がリンク元より後に公開されると、公開までの間そのリンクは本番で404になる。以下を実行し「違反なし」を確認してからpushする。違反が出たらリンク先の公開日をリンク元と同日以前に前倒しする（前倒しした記事自身のリンクが新たな違反を生むことがあるため、違反ゼロになるまで繰り返す）：
   ```bash
   python -X utf8 automation/check-forward-links.py
   ```
3. **内部リンクの末尾スラッシュチェック**：本文の内部リンクがスラッシュ無しで書かれていないか確認する（二重インデックス防止・前述「4. 内部リンクは必ず末尾スラッシュ付きで書く」）。以下を実行し**何も出なければOK**。出たらその記事の該当リンクにスラッシュを足す：
   ```bash
   # markdown記法・HTMLタグ記法の両方で、スラッシュ無しの内部リンクを検出（tag・アンカー・外部URLは対象外）
   # ※ /pricing のようなLPページ直リンク（サブパスなし）も検出対象（2026-07-14修正：旧パターンはサブパス付きしか検出せず273箇所の漏れが発生した）
   grep -rnE '\]\(/(blog|services|cases|demo|pricing|contact)(/[a-z0-9-]+)?\)|href="/(blog|services|cases|demo|pricing|contact)(/[a-z0-9-]+)?"' site/src/content/blog/
   ```

## 8. よくあるトラブル

**Q. 記事を追加したのにサイトに出てこない**
A. まず frontmatter の `date` が未来日付になっていないか確認（予約投稿の仕様）。次に GitHub Actions のビルドが成功しているか確認。

**Q. デプロイが「Deployment failed, try again later.」で失敗した**
A. GitHub側の一時エラー。Actionsの画面で「Re-run **failed** jobs」を実行する（「Re-run **all** jobs」は成果物が二重になり別のエラーが出るため使わない）。それでも直らない場合は空コミットをpushすると新しいビルドが走る。

**Q. 太字にしたはずの箇所に `**` がそのまま表示される**
A. 太字の境界にカギ括弧等がある書き方が原因。「7. 公開前チェック」の太字レンダリング確認を参照。
