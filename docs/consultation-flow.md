# 相談対応フロー・ヒアリングシート

ビズシスのお問い合わせフォームから相談が来たときの対応手順。
誰が受けても「同じ流れ・同じ資料」が出ることをゴールにする。
（判断品質まで揃えるのは無理なので、概算は必ず相互レビューで担保する）

---

## 全体フロー

```
フォーム受信
   ↓ 【1営業日以内】
① 初回返信（テンプレ）＋ヒアリング質問を送付
   ↓
② メールでヒアリング（回答をもとに追加質問・1〜2往復）
   ↓
③ システムイメージ資料（HTML）を作成して認識合わせ
   ↓
④ 概算見積を作成
   ↓ 【必須】
⑤ もう一人がレビュー（概算・実現性の妥当性チェック）
   ↓
⑥ 概算提示・受注判断
```

- フォームで届く情報：氏名／会社名（任意）／メール／相談内容
- 基本は**メールだけで完結**させる。打ち合わせ（オンライン・電話）は、メールが2往復しても要件がつかめない場合の補助手段
- ①は内容を深く検討せず、まず「受け取った」を返すことを優先する
- ⑤を飛ばして概算を送るのは禁止。金額と納期は必ず2人の目を通す

---

## ① 初回返信テンプレ

件名：`【ビズシス】お問い合わせありがとうございます（{氏名}様）`

```
{氏名} 様

お問い合わせいただきありがとうございます。
ビズシスの{担当者名}と申します。

いただいた内容を拝見しました。
状況を正しく理解したうえで進め方や概算の目安をご案内したく、
差し支えない範囲で以下について教えていただけないでしょうか。

1. 現在のやり方
   （Excel・紙・既存システムなど、いま業務をどう回しているか）
2. 一番困っていること・解決したいこと
3. 使う方の人数と、パソコン・スマホどちらで使いたいか
4. ご予算の目安（ざっくりで構いません）
5. 希望される時期・期限があれば

わかる範囲で構いませんし、箇条書きで問題ありません。
文章にしづらい場合は、オンラインやお電話でお伺いすることも可能ですので
その際はお気軽にお申し付けください。

ご返信をお待ちしております。
```

- 質問は5つまで。最初から全部聞こうとしない（残りは②の追加質問で埋める）
- この時点で金額・納期・実現可否には触れない（ヒアリング前の口約束が一番揉める）

## ② ヒアリングシート

メールの回答を受けたら、このシートに沿って「まだ埋まっていない項目」を確認し、
追加質問メールで埋めていく（追加質問も1通あたり5問まで・2往復を目安にする）。
確認できた内容は `output/consultations/{YYYYMMDD}_{会社名 or 氏名}.md` に保存する。

メールで2往復しても要件の輪郭がつかめない場合は、
30分のオンラインまたは電話に切り替える（ずるずるメールを続けない）。

### 1. 現状
- [ ] 事業内容・業務の流れ（どんな仕事で、誰が、何人でやっているか）
- [ ] 今どうやっているか（Excel／紙／既存システム／他社サービス）
- [ ] 既存システムがある場合：作った会社・保守契約の有無・ソースの有無

### 2. 困りごと
- [ ] 一番困っていること（複数出たら優先順位を聞く）
- [ ] それによって起きている実害（時間・ミス・機会損失）
- [ ] なぜ今相談しようと思ったか（きっかけ）

### 3. やりたいこと
- [ ] 実現したいことを具体的に（「効率化したい」で止めず、画面や操作のイメージまで聞く）
- [ ] 使う人は誰か・何人か・ITリテラシーはどの程度か
- [ ] スマホ対応の要否

### 4. 制約
- [ ] 予算感（言いにくそうなら「50万・100万・300万でいうとどのあたりか」と幅で聞く）
- [ ] 希望時期・期限（期限がある場合、その理由も）
- [ ] 連携が必要な外部サービス（LINE・会計ソフト・EC など）
- [ ] データ移行の要否（既存データの件数・形式）

### 5. 進め方
- [ ] 窓口担当者は誰か（決裁者と同一か）
- [ ] 次のアクションの合意（イメージ資料をいつまでに送るか）

## ③ システムイメージ資料

ヒアリング内容をもとに、画面フロー＋モックアップ形式のシステムイメージ資料を作成する。
一覧表形式（Excel／Markdownの機能一覧）は不可。
「提案を多く含む。設計のたたき台として作成」の注記を必ず入れる。

作図ルール：
- **雛型：`docs/templates/システムイメージ雛型.drawio`** を必ず参考にする。
  AIに生成させるときはこのファイルを渡して「同じ作り（凡例・画面ボックス・
  付箋ノート＋点線・順序マーカー）で作成」と指示する。
  顧客・受注管理システムを題材にした実寸サンプルなので、粒度感の基準にもなる
- 補足・注釈は**吹き出し（callout）を使わず、付箋型ノート（drawioの `shape=note`）**で書く
  （吹き出しはしっぽの向き・長さがAI生成で崩れ、手直しの手間が増えるため禁止）
- 注釈と対象箇所は**点線（`endArrow=none;dashed=1;`）でつなぐ**。
  線の端点は付箋側をノードに接続（source指定）、対象側は座標指定にする。
  点線ならスタッフの手直し時にドラッグで付け替えも簡単にできる
- 左寄せの文字は枠線にくっつけず、**`spacingLeft=8`（内側余白）**を必ず指定する
  （先頭に半角スペースを手打ちする方式は、折り返し行に効かないため使わない）
- **スタッフが手直しした後の .drawio にAIで変更を加えるときは、ゼロから再生成せず
  既存ファイルへの部分修正（スタイル置換など）で行う**
  （再生成すると手直しした配置・サイズがすべて消えるため）

### 作成〜お客さんコメントまでの流れ

```
AIが .drawio を生成
   ↓
スタッフが手直し（VS Code拡張「Draw.io Integration」でGUI編集）
   ↓
app.diagrams.net で1ページPDFに出力
   ↓
お客さんへ送付 → Edge でコメントを記入して返送
   ↓
コメントを反映して .drawio を修正 → ver を上げて再送付（1〜2往復）
```

### 各ステップの手順

1. **AI生成**：ヒアリングメモを渡して .drawio 形式で生成させる。
   マスターは `.drawio` ファイル。保存名は `{案件名}システムイメージ_ver1.drawio`
2. **AIセルフチェック**：AIは生成・修正のたびに `automation/drawio_visual_check.py` で
   PNGに描画し、画像を自分で見て以下を確認してから人に渡す
   - オブジェクトや文字が重なっていないか
   - 文字が箱からはみ出したり端に詰まったりしていないか
   - 配置が揃っているか（上下位置・間隔のばらつき）
   - 適切な余白があるか
   - 描画要素数が極端に少なくないか（少ない場合はファイル破損のサイン）
3. **手直し**：VS Code に拡張機能「Draw.io Integration」（hediet.vscode-drawio）を
   インストールしておくと、.drawio を開くだけでGUI編集できる。
   AIセルフチェック後も残る細かな違和感（詰まり・バランス）を人が整える
4. **PDF出力**：https://app.diagrams.net/ で .drawio を開き、
   「ファイル → 形式を指定してエクスポート → PDF...」→ **サイズは「ダイアグラム」を選ぶ**
   （「ページ」のままだとA4で複数ページに分割されて見づらくなるため必須）
   ※VS Code拡張からはPDF出力できない（PNG/SVGのみ）ので、PDF化は必ずapp.diagrams.netで行う
5. **送付**：送付メールに以下の案内を必ず入れる（ChromeのPDFビューアはコメント不可のため）

   ```
   添付のPDFはいったん保存して、Edge で開くとコメントを記入できます。
   （右クリック →「プログラムから開く」→ Microsoft Edge）
   気になる点・認識と違う点を直接書き込んでご返送ください。
   ```

6. **反映**：返ってきたコメントをもとに .drawio を修正し、
   `_ver2` のようにファイル名の版を上げて再出力・再送付する。
   古い版のPDF・drawioも削除せず残す（経緯の証跡になるため）

### 補足

- お客さんが印刷派・PDF注釈に不慣れな場合は、PNG出力してPowerPointに貼った版を
  代わりに送る（スライドサイズを図に合わせれば横長でも1枚に収まる）
- .drawio マスターはお客さんに渡さない（版管理が崩れるため）。渡すのはPDF（またはPPT）のみ

## ④⑤ 概算とレビュー

- 概算は「画面数×工数」ベースでざっくり出し、幅を持たせて提示する（例：80〜120万円）
- 提示前に必ずもう一人へ以下を確認依頼する
  - [ ] 技術的に実現できるか（外部連携・データ移行まわりは特に）
  - [ ] 工数の見落としがないか（認証・権限・CSV入出力・帳票は漏れやすい）
  - [ ] 金額と納期が無理をしていないか
- レビュー結果はヒアリングメモの末尾に追記する

---

## やってはいけないこと

- ヒアリング完了前に金額・納期を答える（メールだと文面が証拠として残るため、口頭以上に危険）
- レビューなしで概算を送る
- 「できます」と即答する（確認して翌営業日までに回答、で問題ない）
- 1通のメールに質問を詰め込みすぎる（5問を超えると返信率が落ちる）
- ヒアリング内容をメモに残さずメール往復だけで進める
