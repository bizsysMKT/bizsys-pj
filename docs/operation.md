# 運用ドキュメント（Operation）

## 設計意図
このドキュメントは、漢那さんが『毎月何をするのか』『困ったときにどうするのか』を一つの場所で調べられるようにするものです。運用開始後の「最初に開くべきドキュメント」が目的です。

## 1. 日常運用タスク＆頻度

### output/ の使い方
- `output/` は成果物ストレージとして扱う。実装完了後のビルド済みサイトや納品資料、スクリーンショットなどを保管し、運用確認やレビュー時の参照に使う。
- 実装担当は `site/` で開発し、必要に応じて `output/` に生成物をコピーしておく。漢那さんやPMは `output/` から成果物を確認できる。

### 毎週のタスク（30分～1時間）

#### 1.1 ブログ記事の追加・更新

**タスク**：週2～3本のブログ記事を追加

**やることリスト**：
1. [ ] Markdown 形式で記事を作成する
2. [ ] GitHub Desktop を開く
3. [ ] 「Create a branch」で新しいブランチを作成（例：blog/04-04-new-article）
4. [ ] docs/blog/ フォルダに `.md` ファイルを保存
5. [ ] GitHub Desktop で「Commit」→「Push」を実行
6. [ ] 2～3分待つ（自動デプロイ中）
7. [ ] bizsys.jp/blog でサイトを開いて『記事が公開されたか』を確認
8. [ ] X に自動投稿されたことを確認（@bizsys_jp のタイムラインを確認）
9. [ ] 問題がなければ「完了」

**記事テンプレート**（使用例も docs/templates/blog_template.md に記載）

```markdown
記事の冒頭（100～150文字）。何について書いている記事かを明確に。

**関連記事**
- [関連記事1](記事URL)
- [関連記事2](記事URL)
```

- **Q. 既に公開した記事を編集したい**  
  A. その `.md` ファイルを編集 → Commit & Push すれば自動反映。

## 2. トラブルシューティング（困ったときの対応）

### 困りごと①：記事を push したのに、サイトに反映されない

**症状**：
- GitHub Desktop で Commit & Push は完了した
- でも 10分経ってもサイトに載っていない

**確認手順**：
1. [ ] GitHub リポジトリの「Actions」タブを開く
2. [ ] 最新のワークフロー実行を確認
   - ✅ **緑色チェック**が表示 → デプロイ成功（単にブラウザキャッシュの問題）
   - ❌ **赤色バツ**が表示 → デプロイ失敗（下記「デプロイエラー」参照）
   - 💛 **黄色●**が表示 → 実行中（もう数分待つ）

**対応**：
- **キャッシュ問題の場合**：
  1. ブラウザの「全ファイルを削除」（Ctrl + Shift + Delete）を実行
  2. bizsys.jp を Ctrl + F5 で強制リロード
  3. 記事が表示されたら OK

- **デプロイエラー（赤色）の場合**：
  1. ワークフロー実行のログを開く（赤色の行をクリック）
  2. エラーメッセージを確認
  3. よくある原因：
     - **Markdown 形式が間違っている** → 括弧や引用符が不正
       → 記事の先頭の `---` 区間（front-matter）を確認
     - **ファイル名が不正**（スペース・特殊文字）
       → ファイル名を英数字 + ハイフンのみに変更
     - **その他**：ログに詳細が書いてあるので Yusuke に報告

---

### 困りごと②：X に自動投稿されない

**症状**：
- 記事を push した
- 記事はサイトに表示された
- でも X に投稿されない

**確認手順**：
1. [ ] GitHub Actions のログを確認（上記「デプロイエラー」と同様）
   - 緑色 → X投稿は成功した（X タイムラインをリロード）
   - 赤色 → 投稿エラー（下記対応）
2. [ ] X (@bizsys_jp) のタイムラインを確認
   - 新しい投稿がある？
3. [ ] Yusuke からのメール受信を確認
   - 「[エラー] X投稿に失敗しました」という件名 → エラー詳細を確認

**対応**：
- **「X API制限に到達」エラー**：
  - X の無料プランは 1時間に 300 投稿まで
  - ビズシスは 1日数本なので通常は大丈夫
  - 短時間に大量投稿した場合は翌日を待つ
  - 対応：投稿スケジュールを分散（Yusuke に依頼）

- **「認証エラー」**：
  - X API キーの有効期限が切れた可能性あり
  - 対応：直ちに Yusuke に報告（キー更新が必要）

- **「API接続不可」**：
  - X のサーバーがダウンしている可能性あり
  - X Status Dashboard を確認：https://status.twitter.com
  - 対応：1～2時間待機。復旧を待つ
  - 同じ記事を手動で投稿したい → 下記「手動投稿手順」へ

---

### 困りごと③：GA4 でアクセス数が 0 件

**症状**：
- 記事を公開した
- サイトにアクセスしてみた
- でも GA4 に計測されていない

**確認手順**：
1. [ ] GA4 ダッシュボードを開く
2. [ ] 「リアルタイムレポート」を確認
   - ページを開いたとき、「アクティブユーザー」に 1 が表示される？
   - YES → 計測 OK（レポート更新に 24時間待つ）
   - NO → トラッキング ID が間違っている可能性

**対応**：
- **リアルタイムで計測されない場合**：
  - ブラウザの「Developer Tools」（F12）を開く
  - Console を確認：エラーメッセージがあるか？
  - あれば Yusuke に報告（GA4 トラッキング ID が誤設定の可能性）

---

### 困りごと④：問い合わせメールが来ない（Formspree）

**症状**：
- HP のお問い合わせフォームから送信してみた
- でも漢那さんのメールに来ない

**確認手順**：
1. [ ] ジャンク・スパムフォルダを確認
2. [ ] Formspree の管理画面で確認：https://formspree.io
   - 「フォーム送信履歴」タブで最新の送信記録を確認
   - 見つかる？ → メール受信設定が間違っている可能性
   - 見つからない？ → フォーム自体が動いていない

**対応**：
- **Formspree に記録がある場合**：
  - Formspree 内で「メール送信先」を確認・変更：
    1. Formspree の設定ページを開く
    2. 「Notifications」で送信先メールアドレスを確認
    3. 変更が必要なら修正 → テスト送信

- **Formspree に記録がない場合**：
  - フォーム自体が壊れている（Markdown 形式エラー等）
  - Yusuke に報告（フォーム修正が必要）

---

### 困りごと⑤：GitHub Actions が突然失敗し始めた

**症状**：
- ここ数日は記事を push すると自動デプロイされていた
- 突然「❌ ワークフロー失敗」になった

**確認手順**：
1. [ ] 最新のワークフロー実行ログを確認
   - エラーメッセージに「ディスク満杯」「メモリ不足」等の表示？
   - → GitHub が明らかにしているエラーなので、原因の見当がつきやすい
2. [ ] 前回成功時との差分を確認
   - 新しい記事を追加した？ファイルが大きすぎない？
   - → ファイルサイズを削減する必要がある可能性

**対応**：
- **一般的なエラー**：ログの内容をメモして Yusuke に報告
- **記事ファイルが大きい**：
  - 画像を圧縮（オンラインツール：tinypng.com など）
  - パワーポイント・大きな PDF の埋め込みは避ける
  - 対応後、再度 Push してください

## 3. 手動対応が必要な場合

### パターンA：記事を急いで公開したい

**例**：記事を明日から公開したいのに GitHub Actions が失敗している

**手順**：
1. [ ] Yusuke に「内容」「URL」「投稿テキスト」を連絡
2. [ ] Yusuke が GitHub Actions を修正
3. [ ] 再度 Push して自動デプロイを試す
4. [ ] それでも失敗した場合：Yusuke が手動でサイトを更新 + X へ投稿

---

### パターンB：X に手動で投稿したい

**例**：自動投稿に失敗、または緊急アナウンスがある

**手順**：
1. [ ] X のアプリ・Web から通常通り @bizsys_jp でツイート
2. [ ] 自動投稿と被らないよう注意

---

### パターンC：記事を非公開にしたい

**例**：間違った内容を公開してしまった

**手順**：
1. [ ] GitHub Desktop で該当の `.md` ファイルを削除
2. [ ] Commit & Push を実行
3. [ ] 2～3分後、サイトから削除される
4. [ ] X への自動投稿は止まりません。手動で当該ツイートを削除してください

---

### パターンD：サイトが完全に落ちた

**例**：ブラウザで bizsys.jp にアクセスできない

**手順**：
1. [ ] GitHub Pages の Status ページを確認
   - URL：https://www.githubstatus.com
   - 「Operational」表示 → GitHub は正常
   - 「Degraded」「Major Outage」表示 → GitHub障害（復旧を待つ）
2. [ ] 自分の GitHub リポジトリの Settings を確認
   - 「Pages」タブを開く
   - デプロイのステータスを確認
3. [ ] 2時間以上復旧しない場合：Yusuke に報告
   - 緊急用代替サーバーへの移行も検討

## 5. よくある質問＆回答

### Q. 記事を書く時間・スキルがない場合は？

**A.** Claude Code を使った記事生成サポートが可能です。
- Yusuke に「キーワード：___」を伝える
- Claude Code が下書きを作成 → 漢那さんが修正・加筆 → 投稿
- 手作業より 50% 以上時間削減できます

---

### Q. アクセス数が目標（50）に到達しない場合は？

**A.** 以下を試してください（優先順）：
1. **SEO対策を強化**：Google Search Console の「検索パフォーマンス」から『クリック数が少ないキーワード』を確認 → 記事の冒頭をそのキーワードを含むように修正
2. **記事の質を上げる**：月次レビューで最高 PV の記事の『何が良かったのか』を分析 → 同じ要素を他記事に応用
3. **投稿頻度を上げる**：週3本 → 週5本に増やす（プロモーション期間）
4. **X投稿のハッシュタグを工夫**：より検索されるキーワード（リサーチツール使用）のハッシュタグを追加

---

### Q. X投稿が失敗する頻度が高い場合は？

**A.** 以下を確認してください：
1. **X API のクォータ確認**：短時間に大量投稿していないか？ → 投稿タイミングを分散
2. **X API キーの有効期限確認**：Yusuke が定期確認（3ヶ月ごと）
3. **X の API 制限**：免除申請が必要なら Yusuke に依頼

---

### Q. 記事の個数を増やしたい（例：週3本 → 週5本）

**A.** 以下の確認が必要：
- GitHub Pages のビルド時間に余裕があるか？ → 試す
- 漢那さんの作業負荷が大丈夫か？ → 週 5本 = 毎日 20分以上必要
- その場合、Claude Code で記事生成補助を導入することをお勧めします

---

### Q. 記事のカテゴリー分けをしたい

**A.** Astro の設計を変更する必要があります（小規模な修正）。Yusuke に依頼してください。
- 作業時間：1ページ分
- 以降、記事の tags を整理するだけで自動分類されます

---

### Q. 記事を複数人で作成したい

**A.** GitHub の アクセス権限を追加する必要があります。
- Yusuke が設定 → チームメンバーが GitHub Desktop で編集 → Commit & Push
- ただし『同時編集』になるとコンフリクト（競合）が起きる可能性 → 担当記事を分けた方が安全
- 方法詳細は Yusuke に相談
