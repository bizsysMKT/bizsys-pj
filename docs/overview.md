# プロジェクト概要書（Overview）

## 設計意図
このドキュメントは、プロジェクト全体像を『ワンページで理解できる』ようにまとめたものです。漢那さんが引き継ぎ時に「このプロジェクトは何をする？どんな構成？」を迷わず答えられるようにすることが目的です。

---

## 1. プロジェクト目的（非エンジニア向け説明）

### 何をするのか
ビズシスの営業活動・集客を『自動化できるWebサイト + SNS + メール』で実現します。

### 従来との違い
- **従来**：ココナラで案件受注、手作業で営業対応
- **新しい方法**：自社Webサイトから集客→自動投稿で継続的に露出→顧客の自然流入

### 最終形
- 自社HP（biz-sys.jp）：24時間営業
- ブログ記事：週2～3本、自動更新
- X（旧Twitter）：毎日自動投稿
- 問い合わせフォーム：自動受信
- すべて漢那さんが一人で運用可能

---

## 2. 対象範囲

### 作る対象
| 区分 | 内容 |
|------|------|
| **Webサイト** | HP（7ページ）+ ブログ（5～10記事） |
| **SNS** | X（@bizsys_jp）自動投稿 |
| **フォーム** | 問い合わせ受信・確認 |
| **分析** | アクセス数・記事閲覧数・成果確認 |
| **運用** | 月1回のレビュー・記事追加・投稿調整 |

### Astro 実装方針
- `site/` 以下を Astro の実装領域とする。
- LP 用ページは `site/src/pages/` に配置。想定ページ一覧は以下。 
  - `/index`
  - `/about`
  - `/services`
  - `/cases`
  - `/tools`
  - `/faq`
  - `/contact`
- ブログは `site/src/pages/blog/` および `site/src/pages/blog/[slug].astro` で構成し、Markdown 記事は `site/src/content/blog/` に保存する。
- `Formspree` 問い合わせフォームは `/contact` ページに掲載し、漢那さんのメールに自動送信される。
- `output/` は公開成果物や配布用ファイルの領域とし、実装作業は `site/` を中心に進める。

### 作らない対象
- Facebook・note・スレッズなど他SNS
- 電話・Zoomなど音声対応
- 細かいSEO対策（基本的なキーワード戦略のみ）
- AIが完全自動で記事を生成・投稿（手動確認が必須）
- システム開発（既存ツール活用）

## 3. フェーズ概要＆スケジュール

| フェーズ | 内容 | 期間目安 | 完了条件 |
|---------|------|---------|---------|
| **フェーズ1** | HPとブログを公開、問い合わせフォーム動作確認、GA4設定 | 2～3週間 | サイト公開・フォーム動作・計測OK |

## 4. 役割分担＆セッション方針

### 実装セッション（今回）
- **PM（監督役）**：各セッション後に「目的達成しているか？」「リスクはないか？」をレビュー
- **設計担当**：このドキュメント群を作成・整理
- **実装担当**：HP・ブログのHTML/CSS制作、公開
- **自動化担当**：GitHub Actions、X投稿、記事生成スクリプト設定

### 運用開始後（漢那さん）
- **所有者**：漢那さん（1人）
- **運用内容**：毎週1～2本記事追加、月1回KPI確認
- **サポート体制**：問題時はYusuke（実装者）へ連絡

## 6. ドキュメント構成

```
docs/
├── project_plan.md        ← プロジェクト計画（上位方針）
├── overview.md            ← このファイル（全体像）
├── architecture.md        ← 技術構成図＆システムの繋がり
├── operation.md           ← 日常運用フロー＋トラブル対応
├── handover.md            ← 引き継ぎチェックリスト
├── phases.md              ← 各フェーズの詳細タスク（任意）
└── templates/
    ├── blog_template.md   ← ブログ記事テンプレート
    └── monthly_report.md  ← 月次レポートテンプレート
```

各ドキュメントの関係：
1. **overview.md**（このファイル）→ プロジェクト全体像を理解
2. **architecture.md** → 「何がどう繋がっているか？」を理解
3. **operation.md** → 「毎月何をするの？トラブル時は？」を理解
4. **handover.md** → 「引き継ぎ前に何をチェック？」を確認
5. **phases.md**（参考） → 「実装セッション内での詳細タスク」

## 7. 非エンジニア向けQ&A

**Q. 毎月どのくらい手間がかかる？**  
A. ブログ記事追加：30分～1時間/週、KPI確認：30分/月。合計 1時間程度/月

**Q. 何かエラーが出たら？**  
A. operation.md の「トラブルシューティング」を見れば対応方法が書いてあります。それでも解決しない場合はYusukeに連絡。

**Q. Webサイトを大きく変更したい場合は？**  
A. architecture.md を見て「何がどう繋がっているか」を理解してから手を加えてください。

**Q. 記事を手動で投稿したい場合は？**  
A. operation.md の「手動投稿手順」を参照。自動での投稿のほか、いつでも手動切り替え可能。
