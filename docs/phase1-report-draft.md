## 4. 振り返り

### 4-1. うまくいったこと

**ロール分離による品質確保**
- PM・設計・実装の責任を明確に分けたことで、各担当が専門性を発揮できた
- コンテンツ仕様（lp-content.md）とデザイン仕様（guidelines.md）を分離したことで、実装担当が迷わず作業できた

**スプリント型レビューサイクル**
- task → report → review の文書サイクルにより、指示・判断・記録が一体で残る
- 後から見返しても経緯が追える。引き継ぎ資料として機能する

**PMの早期課題検出**
- Sprint 3 設計 v2.0 のグラスモーフィズム・連続ダーク問題を実装前に捕捉し、差し戻しコストを1セッション削減できた
- ユーザーの5点指摘を①②③（設計起因）と④⑤（実装対応可能）に切り分け、最小のセッション数で解決できた

---

### 4-2. コストが高かった原因

**① デザイン要件の初期定義不足（Sprint 3 高コストの主因）**

「リッチにしてほしい」というフィードバックを受けてから設計担当が具体化するフローを取ったが、PMが事前に「リッチの定義・取り入れる/取り入れない要素」を言語化しておけば、設計担当の試行錯誤を1〜2ラウンド減らせた可能性がある。

**② 実装後差し戻しの発生**

ヒーローレイアウトの折れ問題・border-radius の過大・アイコン未定義は、設計仕様書に未記載だったために実装後に発覚した。実装担当が仕様の曖昧な部分を自己判断で埋めた結果、意図と異なる実装になった。
