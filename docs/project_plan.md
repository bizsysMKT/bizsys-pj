# プロジェクト計画書

## 1. プロジェクト目的
ビズシスのマーケティング導線（LP / ブログ / SNS）を構築し、自動運用できる仕組みを作り、第三者（漢那さん）へ引き継ぐこと。

プラットフォーム依存脱却（ココナラから自前集客へ）  
Claude Code活用による構築・運用自動化の模索

## 2. スコープ
### やること
- HP構築（GitHub Pages + Astro + Formspree）
- ブログ構築（Astro + markdown管理）
- Xアカウント作成と自動投稿設定（@bizsys_jp）
- 運用自動化（GitHub Actionsデプロイ、記事生成）
- 効果測定（GA4 + Search Console）
- 引き継ぎドキュメント作成

### やらないこと
- 他のSNS（Facebook・note・スレッズ等）の構築
- 電話・Zoom対応（文字ベースのみ）
- 詳細SEO最適化（基本的なSEO意識のみ）
- 外部ツール開発（既存サービス活用）
- 技術選定変更（Astro・GitHub Pages固定）

## 3. フェーズ定義
- フェーズ1：HP＋ブログ公開、Formspree、GA4/Search Console設定
- フェーズ2：GitHub Actions自動デプロイ
- フェーズ3：X自動投稿
- フェーズ4：Claude Code記事生成、操作マニュアル化

**フェーズ間依存関係**：フェーズ1完了が前提条件。その後、フェーズ2→3→4は順次進行する。

## 4. 成功の定義
- HP公開（biz-sys.jp、7ページ構成、問い合わせフォーム動作）
- ブログ記事公開（5〜10本、週2〜3本ペース）
- X自動投稿動作（記事公開→X投稿自動化）
- GA4/Search Consoleデータ取得可能
- 漢那さん引き継ぎ完了（一人で運用可能状態）
- KPI：HP月間50以上、記事平均5以上、自動投稿成功率95%以上（月次レビューで調整可能、エラーハンドリング対応）

### 漢那さん向け確認項目
- 各ツール（GitHub、X、Formspree、GA4、Search Console）のアカウント引き継ぎ困難性の確認
- トラブルシューティング手順書の整備（記事生成失敗時、X投稿失敗時等）
- 月次レビュープロセスの運用体制（実施者、頻度、KPI調整ルール）
- 緊急時の連絡先・エスカレーション方法

## 5. 役割分担
- PM：設計レビュー・意思決定支援・リスク指摘
- 設計担当：docs作成・運用フロー定義
- 実装担当：LP/ブログ実装・公開
- 自動化担当：GitHub Actions・X投稿・記事生成自動化

## 6. 成果物一覧
- docs：overview.md / architecture.md / operation.md / handover.md
- LP：公開HP（biz-sys.jp）
- ブログ：公開ブログ（同ドメイン）
- SNS：Xアカウント（@bizsys_jp）＋自動投稿設定
- automation：GitHub Actionsワークフロー / 記事生成スクリプト

## 7. 前提・制約
- 技術構成固定（GitHub Pages + Astro + Formspree + GA4 + Search Console）
- ディレクトリルール遵守（input/読み取り専用、output/成果物、automation/ロジック、docs/設計）
- 相対パス使用、ユーザーディレクトリ禁止
- ドメイン：biz-sys.jp（漢那さん設定）
- アカウント：GitHub（ユースケ作成）、X（@bizsys_jp新規）、Formspree（漢那メール）、GA4/Search Console（漢那Googleアカウント）

## 8. 主要リスク・対応
| リスク | 影響度 | 対応策 |
|----|----|---------|
| X API仕様変更・サービス停止 | 高 | 自動投稿失敗時の手動切り替え手順を用意。定期的にX API status確認 |
| GitHub Pages・リポジトリ障害 | 高 | バックアップリポジトリの独立保管。障害時の復旧手順書作成 |
| 記事生成品質低下・企画枯渇 | 中 | テンプレート・プロンプト集の整備。月次レビューで品質判定 |
| 自動デプロイ失敗（GitHub Actions） | 中 | エラー通知設定。手動デプロイの手順書作成 |
| 漢那さんの負担増加・運用断絶 | 高 | handover.md等の詳細マニュアル。月1回の定期レビュー体制 |
| GA4・Search Console計測エラー | 低 | 定期的なダッシュボード確認。設定誤りは年1回チェック |
