# スプリント管理ディレクトリ

## 目的
PM（各セッション）と各担当（別セッション）の間で指示・報告・レビューを行うコミュニケーションスペース。

---

## 役割分担ルール（厳守）

| 役割 | 担当すること | 担当しないこと |
|------|------------|--------------|
| **PM** | 要件・フィードバックの言語化、完了基準の定義、品質レビュー、次指示の判断 | 設計の詳細（色・CSS・レイアウト構造）、実装コード |
| **設計担当** | デザイン仕様の策定・ドキュメント化（色・フォント・構造・余白）、ガイドラインの更新 | 実装コードの記述 |
| **実装担当** | 設計仕様に基づくコーディング、ビルド確認、報告 | 設計方針の変更・独自判断によるデザイン変更 |

**PMの指示書（briefing）に書いてよいこと：**
- 達成したいゴール（「リッチに見える」「信頼感がある」など）
- 参考資料・参照先
- 完了基準（何が満たされたら終わりか）
- スコープ（何をやって何をやらないか）

**PMの指示書に書いてはいけないこと：**
- 具体的な CSS 値・クラス名・実装コード
- レイアウト構造の詳細（これは設計担当が決める）
- フォント名・カラーコードの指定（これは設計担当が決める）

---

## スプリントの基本サイクル

```
1. PM → 設計担当 briefing
       ↓（設計担当が report を作成）
2. PM レビュー → 設計成果物を確認
       ↓（GOなら）
3. PM → 実装担当 briefing
       ↓（実装担当が report を作成）
4. PM レビュー → 実装成果物を確認
       ↓（GOなら）
5. 次スプリントへ
```

設計フェーズが不要なスプリント（コンテンツ修正のみ等）は手順1〜2を省略できる。  
ただし「デザインに関わる変更」が含まれる場合は必ず設計フェーズを経ること。

## ディレクトリ構成

```
docs/sprint/
├── README.md                  ← このファイル（ルール・使い方ガイド）
├── sprint-00/                 ← Sprint 0：仕切り直し・準備（完了）
├── sprint-01/                 ← Sprint 1：LPコンテンツ実装（完了）
├── sprint-02/                 ← Sprint 2：デザイン・スタイリング（完了）
├── sprint-03/                 ← Sprint 3：リッチデザイン化（進行中）
└── sprint-04/                 ← Sprint 4：全ページ展開・公開準備（未着手）
```

## 各スプリントのファイル

| ファイル | 作成者 | 内容 |
|---------|-------|------|
| `task-for-design.md` | PM | 設計担当へのタスク指示（デザインフェーズあるスプリントのみ） |
| `report-design.md` | 設計担当 | 設計作業の完了報告・PM確認依頼 |
| `review-design.md` | PM | 設計成果物へのレビュー・GO判断 |
| `briefing.md` | PM | 実装担当へのタスク指示 |
| `report.md` | 実装担当 | 実装完了報告・PM確認依頼 |
| `review.md` | PM | 実装成果物へのレビュー・Sprint完了判断 |

## セッション起動時の確認事項

**設計担当セッション**：`task-for-design.md` を読んでから作業開始  
**実装担当セッション**：`briefing.md` を読んでから作業開始（設計フェーズがある場合は `review-design.md` も読む）  
**PM（次セッション）**：最新スプリントの `report.md` または `report-design.md` を読んでレビュー開始
